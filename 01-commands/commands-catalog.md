# 01 – Команды

Этот файл является каталогом команд, режимов и управленческих инструкций, которые используются в рамках курса и практической работы с Claude Code и смежными ИИ-инструментами.

Сюда вынесены базовые команды, которые нужны для:
– запуска сессии;
– управления контекстом;
– очистки и возобновления диалога;
– диагностики;
– проверки permissions;
– понимания того, как именно Claude Code работает в проекте.

Главный принцип раздела: каждая команда оформляется как самостоятельная карточка с понятным ID, назначением, синтаксисом, примером, ошибками и проверкой результата.

---

## Карточка команды

### ID: CMD-START-001
**Название:** Запуск Claude Code из директории проекта

**Когда использовать:**  
Используется в самом начале работы, когда нужно открыть проект в Claude Code и стартовать сессию в контексте конкретной папки.

**Цель:**  
Запустить Claude Code так, чтобы он работал именно с текущим проектом, видел его файлы и мог анализировать структуру репозитория.

**Входные данные:**  
– путь к директории проекта;  
– установленный Claude Code;  
– активный логин, если он уже выполнен.

**Синтаксис:**  
```bash
cd /path/to/your/project
claude
```

**Пример:**

```bash
cd ~/Desktop/test-project
claude
```

**Антипример:**
Запускать Claude Code из случайной директории, не относящейся к проекту, а потом ожидать, что он будет корректно понимать структуру нужного репозитория.

**Типовые ошибки:**
– пользователь запускает Claude Code не из той папки;
– проект открыт локально, но команда запускается из домашней директории;
– ожидается, что Claude сам найдёт нужный проект без перехода в его каталог.

**Проверка результата:**
После запуска должна стартовать интерактивная сессия Claude Code в контексте текущего проекта. На старте можно попросить кратко объяснить, о чём проект и какие ключевые файлы он видит.

**Связанные разделы:**
– CMD-HELP-001
– CMD-CONTEXT-001
– 00-README/README.md
– PLAYBOOK.md

### ID: CMD-HELP-001

**Название:** /help

**Когда использовать:**
Когда нужно быстро увидеть доступные slash-команды и понять, чем можно управлять внутри текущей сессии.

**Цель:**
Открыть встроенную справку по доступным командам Claude Code.

**Входные данные:**
Активная интерактивная сессия Claude Code.

**Синтаксис:**

```text
/help
```

**Пример:**
Пользователь запускает Claude Code, забывает точное название команды для очистки истории и вызывает:

```text
/help
```

**Антипример:**
Пытаться вспомнить все команды по памяти и работать вслепую, когда встроенная справка уже есть в системе.

**Типовые ошибки:**
– пользователь ожидает, что /help решит проблему с конфигами или permissions;
– команда вызывается слишком поздно, хотя могла сразу показать нужный набор инструментов.

**Проверка результата:**
После вызова должен появиться список доступных команд с краткими пояснениями.

**Связанные разделы:**
– CMD-CLEAR-001
– CMD-RESUME-001

### ID: CMD-CLEAR-001

**Название:** /clear

**Когда использовать:**
Когда текущий диалог засорён старым контекстом, модель начинает отвечать не по теме или нужно полностью начать новый разговор внутри текущей сессии.

**Цель:**
Полностью очистить текущую историю диалога и освободить контекст.

**Входные данные:**
Активная сессия Claude Code, в которой накопился лишний контекст.

**Синтаксис:**

```text
/clear
```

**Пример:**
Ты обсуждал отладку тестов, потом резко переключился на MCP, и Claude продолжает тащить старую тему. В этом случае используешь:

```text
/clear
```

**Антипример:**
Использовать /clear, когда тебе важно сохранить ход рассуждения и контекст текущей задачи.

**Типовые ошибки:**
– очищают диалог слишком рано и теряют полезный контекст;
– путают /clear с /compact;
– ожидают, что /clear удалит настройки или память, хотя он очищает историю разговора.

**Проверка результата:**
После выполнения новая тема должна начинаться с чистого контекста, а старый диалог не должен влиять на ответы.

**Связанные разделы:**
– CMD-COMPACT-001
– CMD-CONTEXT-001
– DIAG-CONTEXT-001

### ID: CMD-COMPACT-001

**Название:** /compact

**Когда использовать:**
Когда нужно сократить историю диалога, но не потерять её полностью.

**Цель:**
Очистить историю разговора, сохранив сжатое резюме в контексте.

**Входные данные:**
Длинная активная сессия, в которой нужно уменьшить расход контекста.

**Синтаксис:**

```text
/compact
```

**Дополнительный вариант:**

```text
/compact [instructions for summarization]
```

**Пример:**
Если диалог уже длинный, но ты не хочешь полностью сбрасывать рабочую нить, вместо /clear используешь:

```text
/compact
```

**Антипример:**
Использовать /compact, когда нужно полностью убрать влияние старой темы без сохранения краткой выжимки.

**Типовые ошибки:**
– ожидают полный сброс, а получают сжатое сохранение контекста;
– не проверяют, что именно осталось в summary;
– используют /compact слишком поздно, когда контекст уже почти переполнен.

**Проверка результата:**
После выполнения история должна сократиться, а у Claude должен остаться сжатый смысл предыдущего обсуждения.

**Связанные разделы:**
– CMD-CLEAR-001
– CMD-CONTEXT-001
– DIAG-CONTEXT-001

### ID: CMD-RESUME-001

**Название:** /resume

**Когда использовать:**
Когда нужно вернуться к одному из прошлых диалогов внутри Claude Code и продолжить его.

**Цель:**
Открыть список предыдущих разговоров и продолжить выбранную сессию.

**Входные данные:**
Наличие прошлых диалогов в истории Claude Code.

**Синтаксис:**

```text
/resume
```

**Пример:**
Ты работал вчера над README, а сегодня хочешь продолжить именно тот разговор:

```text
/resume
```

**Антипример:**
Использовать /resume, если тебе нужен полностью новый диалог без возврата к старой ветке.

**Типовые ошибки:**
– пользователь думает, что старый контекст восстановится абсолютно идеально;
– ожидает, что /resume заново прогрузит всё как в первый момент без потерь;
– продолжает старую сессию, хотя логичнее начать новую.

**Проверка результата:**
После вызова должен появиться список прошлых сессий, из которого можно выбрать нужную.

**Связанные разделы:**
– CMD-CLI-RESUME-001
– CMD-CLEAR-001
– CMD-CONTEXT-001

**Проверка результата:**

### ID: CMD-CONTEXT-001

**Название:** /context

**Когда использовать:**
Когда нужно проверить текущее состояние контекста, понять, сколько токенов занято, какие memory-файлы уже подключены и сколько осталось свободного пространства.

**Цель:**
Дать прозрачную картину загрузки контекстного окна.

**Входные данные:**
Активная сессия Claude Code.

**Синтаксис:**

```text
/context
```

**Пример:**
Перед большой задачей или после длинного диалога пользователь проверяет:

```text
/context
```

**Антипример:**
Игнорировать состояние контекста и удивляться, почему Claude начинает терять ранние куски информации.

**Типовые ошибки:**
– проверяют контекст слишком редко;
– замечают переполнение слишком поздно;
– путают общий объём контекста с содержимым memory-файлов.

**Проверка результата:**
В выводе должно быть видно:
– общее использование контекста;
– свободное место;
– memory files;
– распределение токенов по системным блокам, сообщениям и другим источникам.

**Связанные разделы:**
– CMD-CLEAR-001
– CMD-COMPACT-001
– DIAG-CONTEXT-001

### ID: CMD-DOCTOR-001

**Название:** /doctor

**Когда использовать:**
Когда Claude Code ведёт себя странно, настройки работают не так, как ожидалось, или нужно проверить корректность конфигов.

**Цель:**
Провести встроенную диагностику установки и настроек Claude Code.

**Входные данные:**
Активная сессия Claude Code.

**Синтаксис:**

```text
/doctor
```

**Пример:**
Если permissions настроены, но не срабатывают как нужно, запускаешь:

```text
/doctor
```

**Антипример:**
Искать проблему вручную в десятках файлов до запуска встроенной диагностики.

**Типовые ошибки:**
– используют regex-подобные паттерны там, где ожидается префиксное совпадение;
– не запускают /doctor после правок settings.json;
– думают, что проблема в модели, хотя она в конфиге.

**Проверка результата:**
Команда должна показать ошибки конфигурации, невалидные паттерны и подозрительные настройки.

### ID: CMD-PERMISSIONS-001

**Название:** /permissions

**Когда использовать:**
Когда нужно посмотреть текущие правила allow, ask, deny и понять, какие действия разрешены автоматически, какие требуют подтверждения, а какие запрещены.

**Цель:**
Проверить и визуально просмотреть активные permission-правила в рабочей сессии.

**Входные данные:**
Активная сессия Claude Code с настроенными permissions.

**Синтаксис:**

```text
/permissions
```

**Пример:**
После редактирования `.claude/settings.json` пользователь проверяет:

```text
/permissions
```

**Антипример:**
Предполагать, что правило точно применилось, не проверяя его фактическое отображение в интерфейсе Claude Code.

**Типовые ошибки:**
– правки внесены, но сессия не была перезапущена;
– пользователь смотрит только allow и забывает про ask/deny;
– правило сформулировано с ошибкой и не матчится как ожидалось.

**Проверка результата:**
В интерфейсе должны отображаться вкладки Allow, Ask, Deny и Workspace, а нужные правила должны быть видны в соответствующих секциях.

**Связанные разделы:**
– CMD-DOCTOR-001
– CMD-PERM-RULES-001
– CMD-PERM-BASH-001

### ID: CMD-PERM-RULES-001

**Название:** Правила allow, ask, deny

**Когда использовать:**
Когда нужно проектировать безопасный и удобный режим работы Claude Code.

**Цель:**
Понять базовую логику permissions и правильно распределять действия по уровням контроля.

**Входные данные:**
Файл настроек permissions или задача по проектированию режима безопасности.

**Синтаксис или структура:**

```json
{
  "permissions": {
    "allow": ["Bash(npm:*)", "Bash(git:*)"],
    "ask": ["Bash(git push:*)"],
    "deny": ["Bash(rm -rf:*)", "Read(.env)", "Read(.env.*)"]
  }
}
```

**Пример:**
– allow: безопасные и частые действия выполняются автоматически;
– ask: чувствительные действия требуют ручного подтверждения;
– deny: опасные действия полностью блокируются.

**Антипример:**
Сваливать все действия в allow ради скорости и при этом ждать безопасного поведения по умолчанию.

**Типовые ошибки:**
– забывают про приоритет правил;
– считают, что allow всегда победит;
– не отделяют опасные команды от рутинных.

**Проверка результата:**
Нужно убедиться, что:
– безопасные команды проходят автоматически;
– чувствительные команды требуют confirm;
– опасные действия блокируются.

**Связанные разделы:**
– CMD-PERM-PRIORITY-001
– CMD-PERM-BASH-001
– DIAG-PERMISSIONS-001

### ID: CMD-PERM-PRIORITY-001

**Название:** Приоритет правил permissions

**Когда использовать:**
Когда одно и то же действие может совпасть сразу с несколькими правилами.

**Цель:**
Понять, какое правило сработает первым.

**Входные данные:**
Набор allow, ask, deny с потенциально пересекающимися паттернами.

**Синтаксис или структура:**
Приоритет правил:

```text
Deny > Ask > Allow
```

**Пример:**

```json
{
  "permissions": {
    "allow": ["Bash(echo:*)"],
    "deny": ["Bash(echo secret:*)"]
  }
}
```

В этом случае общий `echo` разрешён, но `echo secret ...` должен блокироваться.

**Антипример:**
Считать, что если команда попала в allow, то deny уже не имеет значения.

**Типовые ошибки:**
– забывают, что deny сильнее всего;
– неверно проектируют исключения;
– не тестируют пересечения правил.

**Проверка результата:**
Нужно проверить реальное поведение команды и убедиться, что пересечение правил работает по порядку: deny, затем ask, затем allow.

**Связанные разделы:**
– CMD-PERM-RULES-001
– CMD-PERM-BASH-001
– CMD-DOCTOR-001

### ID: CMD-PERM-BASH-001

**Название:** Bash-паттерны и префиксное совпадение

**Когда использовать:**
Когда нужно разрешать, спрашивать или запрещать shell-команды по шаблону.

**Цель:**
Правильно задавать Bash-правила без невалидных или нестабильных шаблонов.

**Входные данные:**
Bash-команды, которые нужно контролировать через permissions.

**Синтаксис или структура:**
Используется префиксный матчинг вида:

```json
{
  "permissions": {
    "allow": ["Bash(git:*)"]
  }
}
```

**Пример:**

```json
{
  "permissions": {
    "allow": ["Bash(npm:*)", "Bash(git:*)"],
    "ask": ["Bash(git push:*)"],
    "deny": ["Bash(rm -rf:*)"]
  }
}
```

**Антипример:**
Использовать regex-подобные конструкции там, где ожидается простой префиксный матчинг.

**Типовые ошибки:**
– вместо `:*` используют просто `*`;
– пытаются применять regex и получают нестабильное поведение;
– забывают проверить правило через /doctor и /permissions.

**Проверка результата:**
После настройки Bash-паттерны должны корректно матчить нужные команды, а диагностика не должна показывать ошибок.

**Связанные разделы:**
– CMD-DOCTOR-001
– CMD-PERMISSIONS-001
– CMD-PERM-RULES-001

### ID: CMD-PERM-ADDITIONAL-DIRS-001

**Название:** additionalDirectories и /add-dir

**Когда использовать:**
Когда Claude Code нужно дать доступ к файлам вне текущей директории проекта, например в монорепе, shared-библиотеках или соседних рабочих каталогах.

**Цель:**
Расширить область доступа Claude Code за пределы корня текущего проекта.

**Входные данные:**
Список внешних директорий, к которым реально нужен доступ.

**Синтаксис или структура:**

```json
{
  "permissions": {
    "additionalDirectories": ["../shared-libs/", "~/dotfiles/"]
  }
}
```

**Пример:**
Если часть общей логики лежит в `../shared-libs/`, эту папку можно явно добавить в список дополнительных директорий.

**Антипример:**
Открывать слишком широкий доступ ко всей файловой системе без реальной необходимости.

**Типовые ошибки:**
– добавляют слишком много директорий;
– забывают про рост рисков безопасности;
– не документируют, зачем был открыт внешний доступ.

**Проверка результата:**
Claude Code должен видеть и читать только те внешние директории, которые добавлены явно.

**Связанные разделы:**
– CMD-PERMISSIONS-001
– 03-mcp/mcp-configs-patterns.md
– 05-troubleshooting/faq-diagnostics.md

## Мини-чеклист по разделу команд

Перед тем как считать, что команда понята и оформлена правильно, проверь:
– ясно ли, когда её использовать;
– есть ли у неё конкретный пример;
– описан ли антипример;
– зафиксированы ли типовые ошибки;
– понятно ли, как проверить результат;
– связана ли карточка с другими разделами базы знаний.
