# Contributing to Prompt Ops Playbook

## Как внести вклад

### Сообщить о проблеме
1. Проверьте существующие issues
2. Создайте новый issue с описанием
3. Укажите:
   - Что не так
   - Как воспроизвести
   - Ожидаемое поведение
   - Скриншоты (если применимо)

### Предложить улучшение
1. Опишите проблему или идею
2. Предложите решение
3. Обсудите в issues

### Добавить контент

#### Структура Knowledge Card
```markdown
# Title

## Тип
Concept | Procedure | Command | Config | Pattern | Pitfall | Integration

## Что это
2-4 предложения описания

## Когда применять
Контекст использования

## Минимальный пример
Код/команда/шаги

## Расширение
Дополнительные возможности

## Ошибки и ограничения
Что может пойти не так

## Источник
Ссылка на файл/секцию
```

#### Где размещать
| Тип | Папка |
|-----|-------|
| CLI команды | 01-commands/ |
| Навыки | 02-skills/ |
| MCP | 03-mcp/ |
| Субагенты | 04-subagents/ |
| CodeX | 09-codex/ |

### Pull Request Process

1. **Fork** репозиторий
2. **Создайте ветку**:
   ```bash
   git checkout -b feature/my-feature
   # или
   git checkout -b fix/issue-description
   ```

3. **Внесите изменения**:
   - Следуйте структуре
   - Добавьте источники
   - Проверьте форматирование

4. **Проверьте**:
   - Все ссылки работают
   - Нет опечаток
   - Форматирование корректно

5. **Создайте PR**:
   - Опишите изменения
   - Укажите связанные issues
   - Добавьте примеры

### Стандарты

#### Markdown
- Заголовки: `# ## ###`
- Код-блоки с указанием языка
- Таблицы для структурированных данных
- Максимум 100 символов в строке

#### Источники
- Указывайте источник для каждого факта
- Формат: `https://github.com/owner/repo`
- Для цитат: файл и строка

#### Лицензии
- MIT для кода
- CC-BY-SA для документации
- Указывайте авторство

### Code of Conduct

- Будьте уважительны
- Конструктивная критика
- Фокус на содержании, не на личности
- Помогайте новичкам

### Вопросы?

Создайте issue с меткой `question`.
